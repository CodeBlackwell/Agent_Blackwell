## Overview

Agent Blackwell is an agent-based orchestration system that coordinates multiple AI agents to handle complex tasks through a Redis-based message queue system. The orchestrator manages task distribution, subtask routing, and result aggregation across different specialized agents.

### Key Components

- **Orchestrator**: Central coordinator that manages task flow and agent communication
- **Agent Registry**: Manages registration and discovery of available agents
- **Redis Streams**: Message queue system for task distribution and result collection
- **Vector Storage**: Optional Pinecone integration for memory and context storage
- **FastAPI**: REST API for external task submission and monitoring

## Prerequisites

- Python 3.9 or higher
- Redis server (local or remote)
- Poetry for dependency management
- Optional: Pinecone account for vector storage
- Optional: OpenAI API key for agent functionality

## Configuration

### Orchestrator Configuration

The orchestrator can be configured with various parameters:

```python
from src.orchestrator.main import Orchestrator

# Basic configuration
orchestrator = Orchestrator(
    redis_url="redis://localhost:6379",
    task_stream="agent_tasks",
    result_stream="agent_results",
    is_test_mode=False,
    openai_api_key="your_key_here",
    pinecone_api_key="your_key_here",
    skip_pinecone_init=False
)
```

### Environment Modes

#### Test Mode (`is_test_mode=True`)
- Uses `test_` prefixed streams
- Simplified routing for testing
- No agent-specific stream publishing
- Ideal for unit tests and development

#### Production Mode (`is_test_mode=False`)
- Uses `agent_tasks` and `agent_results` streams
- Publishes to both main and agent-specific streams
- Full routing and redundancy
- Recommended for production deployments

## Basic Usage

### 1. Initialize the Orchestrator

```python
import asyncio
from src.orchestrator.main import Orchestrator

async def main():
    # Initialize orchestrator
    orchestrator = Orchestrator(
        redis_url="redis://localhost:6379",
        is_test_mode=False
    )

    # Initialize agents (if using agent registry)
    await orchestrator.initialize_agents()
```

### 2. Enqueue Tasks

```python
# Enqueue a specification task
task_data = {
    "user_request": "Create a web application for task management",
    "context": "User wants a simple todo app with authentication",
    "requirements": ["user authentication", "task CRUD", "responsive design"]
}

task_id = await orchestrator.enqueue_task("spec", task_data)
print(f"Task enqueued with ID: {task_id}")
```

### 3. Process Tasks (Main Loop)

```python
# Start the orchestrator processing loop
await orchestrator.run_loop()
```

### 4. Register Custom Agents

```python
from your_agents import CustomAgent

# Register a custom agent
custom_agent = CustomAgent()
orchestrator.register_agent("custom", custom_agent)
```

## API Reference

### Orchestrator Class

#### Constructor Parameters

- `redis_url` (str): Redis connection URL
- `task_stream` (str): Main task stream name
- `result_stream` (str): Result stream name
- `is_test_mode` (bool): Enable test mode
- `openai_api_key` (str, optional): OpenAI API key
- `pinecone_api_key` (str, optional): Pinecone API key
- `skip_pinecone_init` (bool): Skip Pinecone initialization

#### Key Methods

##### `async enqueue_task(task_type: str, task_data: Dict[str, Any]) -> str`
Enqueue a task for processing.

**Parameters:**
- `task_type`: Type of task (spec, design, coding, review, test)
- `task_data`: Dictionary containing task information

**Returns:** Task ID string

##### `async process_task(task: Dict[str, Any]) -> Dict[str, Any]`
Process a single task through the appropriate agent.

##### `register_agent(agent_name: str, agent_executor) -> None`
Register an agent with the orchestrator.

##### `async initialize_agents() -> None`
Initialize all registered agents.

##### `async run_loop() -> None`
Start the main processing loop.

### Supported Task Types

- `spec` / `spec_agent`: Specification and requirements analysis
- `design` / `design_agent`: System design and architecture
- `coding` / `coding_agent`: Code implementation
- `review` / `review_agent`: Code review and quality assurance
- `test` / `test_agent`: Testing and validation

### Agent Stream Mapping

| Task Type | Streams |
|-----------|---------|
| spec | `agent:spec:input`, `agent:spec_agent:input` |
| design | `agent:design:input`, `agent:design_agent:input` |
| coding | `agent:coding:input`, `agent:coding_agent:input` |
| review | `agent:review:input`, `agent:review_agent:input` |
| test | `agent:test:input`, `agent:test_agent:input` |

### Response Format

All API responses follow this structure:

```json
{
  "message": "✅ Your request has been queued. Task ID: abc123-def456-ghi789",
  "task_id": "abc123-def456-ghi789",
  "status": "queued"
}
```

Task status responses include:

```json
{
  "task_id": "abc123-def456-ghi789",
  "status": "completed",
  "progress": 100,
  "created_at": "2024-01-15T10:30:00Z",
  "updated_at": "2024-01-15T10:35:00Z",
  "result": {
    "output": "Generated specification document...",
    "artifacts": ["spec.md", "requirements.json"]
  }
}
```

### Error Handling

The API returns appropriate HTTP status codes:

- `200`: Success
- `422`: Validation error (invalid request format)
- `404`: Task not found
- `500`: Internal server error

Example error response:
```json
{
  "detail": "Invalid command format. Commands should start with '!' (e.g., '!deploy')"
}
```

### Direct Feature Request API

Submit feature requests directly to the spec agent without using ChatOps format:

```bash
# Create a new feature request
curl -X POST "http://localhost:8000/api/v1/chatops/feature" \
  -H "Content-Type: application/json" \
  -d '{
    "description": "Add dark mode support to the dashboard",
    "priority": "high",
    "tags": ["ui", "enhancement"],
    "context": { "affected_components": ["frontend", "settings"] }
  }'
```

**Request Fields:**
- `description` (required): Description of the feature request
- `priority` (optional): Priority level (high, medium, low)
- `tags` (optional): List of tags for categorization
- `context` (optional): Additional contextual information as key-value pairs

**Response Format:**

Success response (HTTP 200):
```json
{
  "task_id": "ca9f4db5-7ab0-4238-8316-616a2cdb2c72",
  "message": "Feature request submitted successfully",
  "status_endpoint": "/api/v1/chatops/status/ca9f4db5-7ab0-4238-8316-616a2cdb2c72"
}
```

Server error (HTTP 500):
```json
{
  "detail": "Failed to create feature request: error message"
}
```

### Direct Task Status API

Get task status directly without using ChatOps format:

```bash
# Get task status by task ID
curl -X GET "http://localhost:8000/api/v1/chatops/status/your-task-id-here"

# Example with actual task ID
curl -X GET "http://localhost:8000/api/v1/chatops/status/ca9f4db5-7ab0-4238-8316-616a2cdb2c72"
```

**Response Format:**

Success response (HTTP 200):
```json
{
  "task_id": "ca9f4db5-7ab0-4238-8316-616a2cdb2c72",
  "status": "completed",
  "progress": 100.0,
  "result": {
    "output": "Generated microservices architecture design...",
    "artifacts": ["architecture.md", "services.json"]
  },
  "created_at": "2024-01-15T10:30:00Z",
  "updated_at": "2024-01-15T10:35:00Z"
}
```

Task not found (HTTP 404):
```json
{
  "detail": "Task with ID invalid-task-id not found"
}
```

Server error (HTTP 500):
```json
{
  "detail": "An unexpected error occurred. Please try again later."
}
```

**Status Values:**
- `"queued"` - Task is waiting to be processed
- `"running"` - Task is currently being executed
- `"completed"` - Task finished successfully
- `"failed"` - Task encountered an error
- `"unknown"` - Status could not be determined
